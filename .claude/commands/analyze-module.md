## Оценка качества существующих тестов

Для КАЖДОГО теста в spec-файле определи уровень глубины:

### Shallow (поверхностный) — НЕ считается реальным покрытием:

- Только `toBeVisible()` / `toHaveURL()` без проверки данных
- `expect(typeof x).toBe('number')` или `toBe('boolean')` — всегда true
- Навигация → проверка URL → конец
- Только проверка наличия элемента без взаимодействия

### Medium (средний) — частичное покрытие:

- Клик по элементу + проверка результата клика
- Проверка текста в конкретном элементе
- Заполнение формы (без проверки отправки)
- Переключение табов + проверка отображения

### Deep (глубокий) — реальное покрытие:

- CRUD workflow: создание → проверка → редактирование → удаление
- Проверка данных в таблице (значения, формат, суммы)
- Фильтрация + проверка что ВСЕ строки соответствуют фильтру
- Негативные сценарии: невалидные данные → ошибка
- Проверка прав: действие без прав → отказ
- Межмодульная проверка: действие в модуле A → результат в модуле B
- Проверка API response body (не только status code)

### Метрика качества:

```
Качество = (medium × 0.5 + deep × 1.0) / total × 100%
```

- **< 30%** — критически низкое качество, почти всё нужно переписать
- **30-50%** — низкое, нужны глубокие тесты для P0/P1
- **50-70%** — среднее, добавить workflow и негативные тесты
- **> 70%** — хорошее качество

## Перекрёстный анализ

Сравни ВСЕ источники между собой:

- Код ↔ БД: поля есть в модели, но не в таблице (или наоборот)
- Код ↔ Confluence+Figma: требование описано, но не реализовано
- Код ↔ Figma: элемент в макете, но не в UI
- Код ↔ Код: обработка ошибок, null-проверки, permissions
- GitLab ↔ Тесты: тикет без автотеста, баг без регрессионного теста
- E2E ↔ API: фича покрыта API-тестом, но нет UI-теста
- **Тесты ↔ Бизнес-логика**: какие бизнес-правила НЕ проверяются автотестами

5. Пробелы в покрытии — **не только количество, но и качество**
6. Рекомендации: что делать дальше
