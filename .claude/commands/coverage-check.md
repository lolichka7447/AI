Проверь покрытие тестами для модуля $ARGUMENTS:

## 1. Собери тест-кейсы

**Источники (в порядке приоритета):**

- Файл `_qa-report-$ARGUMENTS.md` (если существует)
- Confluence MCP `atlassian` → найди тест-кейсы для модуля
- Если ничего нет — предложи сначала запустить `/analyze-module $ARGUMENTS`

## 2. Найди существующие автотесты

Ищи в директориях:

- `tests/e2e/` — UI-тесты (Playwright TypeScript)
- `tests/api/` — API-тесты
- `tests/db/` — DB-тесты
- `_target-project/**/src/test/` — unit/integration тесты

Для каждого найденного теста определи:

- Какой тест-кейс он покрывает (по ID или описанию)
- Тип теста (unit, integration, e2e, api)
- Статус: проходит / падает / отключён
- **Глубина: shallow / medium / deep** (см. критерии ниже)

## 3. Оцени ГЛУБИНУ каждого теста

### Классификация глубины:

**Shallow (поверхностный) — вес 0.0:**

- Только `toBeVisible()`, `toHaveURL()`, `toBeAttached()`
- `expect(typeof x).toBe('number')` — всегда true, бесполезно
- `expect(typeof isVisible).toBe('boolean')` — всегда true
- Навигация → ожидание загрузки → конец (без assertions на данные)
- Проверка что элемент существует, без взаимодействия с ним

**Medium (средний) — вес 0.5:**

- Клик → проверка видимости результата
- Заполнение формы (без проверки submit-результата)
- Переключение табов → проверка отображения нового контента
- Проверка текста конкретного элемента
- Сортировка → проверка что первый элемент изменился

**Deep (глубокий) — вес 1.0:**

- CRUD: создание → проверка в таблице → редактирование → удаление
- Проверка данных: значения ячеек, суммы, форматы
- Фильтрация → проверка что ВСЕ строки соответствуют критерию
- Негативные: невалидный ввод → сообщение об ошибке
- Права: действие без прав → запрет
- Workflow: действие в модуле A → проверка результата в модуле B
- API: проверка response body полей (не только status code)
- Данные: экспорт → проверка содержимого файла

### Метрики:

```
Эффективное покрытие = (medium × 0.5 + deep × 1.0) / total × 100%
Реальный рейтинг = эффективное_покрытие × (покрытые_фичи / все_фичи)
```

## 4. Получи тикеты из GitLab

**Если подключён MCP `gitlab`:**

- Получи issues из `noveo-internal-tools/ttt-spring` для модуля $ARGUMENTS
- Фильтруй по labels: `$ARGUMENTS`, `bug`, `feature`, `enhancement`
- Определи requirements из открытых issues
- Сравни milestones с текущим покрытием

**Для каждого тикета определи:**

- Есть ли автотест (E2E или API) покрывающий эту функциональность
- Если баг — есть ли регрессионный тест
- Если feature — покрыта ли функциональность

## 6. Выведи приоритизированный отчёт

1. **Общая статистика**: X из Y тест-кейсов автоматизированы
   - E2E покрытие: A% (из них shallow: N%, medium: M%, deep: K%)
   - API покрытие: B%
   - **Эффективное покрытие: C%** (с учётом глубины)
   - Общий рейтинг качества: X/10
2. **Критические пробелы** — P0 фичи без deep-тестов
3. **Shallow тесты для переписывания** — список тестов с глубиной shallow, которые нужно углубить
4. **Непокрытые P0**: КРИТИЧНО — список тест-кейсов без автотестов
5. **Непокрытые P1**: ВАЖНО — список
6. **Только API (нет E2E)**: список фич покрытых API-тестами, но без UI-тестов
7. **Тикеты без тестов**: GitLab issues без привязанных автотестов
8. **Рекомендация**: предложи запустить `/generate-tests $ARGUMENTS` для непокрытых P0/P1, указав что генерировать глубокие тесты
9. **Тренд**: если есть `reports/history.json`, покажи тренд pass rate для модуля (последние 5 запусков). Предложи `/quality-trend $ARGUMENTS` для детального анализа
