Проанализируй модуль $ARGUMENTS используя QA Expert System:

## Сбор данных (параллельно)

1. **Кодовая база** — найди в репозитории (или `_target-project/`):
   - Контроллеры/эндпоинты модуля с HTTP-методами и путями
   - Сервисы и бизнес-логику
   - Entity/модели и их валидации
   - Обработку ошибок и security-конфиг
   - Существующие тесты для этого модуля

2. **Confluence** — если подключён MCP `atlassian` или есть ссылка в `.env`:
   - Найди требования и user stories для модуля
   - Найди acceptance criteria
   - Если MCP недоступен — попроси пользователя скопировать текст

3. **Figma** — если есть ссылка в `.env` (FIGMA_URL):
   - Открой макеты через Playwright MCP или Figma MCP
   - Выяви все состояния UI: default, hover, error, loading, empty, disabled

4. **БД** — если есть connection string в `.env`:
   - Получи схему таблиц модуля (через MCP `postgres` или скрипт `_db-schema-dump.js`)
   - Сравни с Entity-классами в коде

5. **GitLab тикеты** — если подключён MCP `gitlab`:
   - Получи issues из `noveo-internal-tools/ttt-spring` для модуля $ARGUMENTS
   - Открытые issues — требования и баги
   - Issues с labels `bug`, `feature`, `enhancement`
   - Merge Requests — недавние изменения в модуле
   - Milestones — scope текущего релиза

6. **Существующие тест-кейсы** — проанализируй покрытие:
   - E2E тесты в `tests/e2e/` — найди spec-файл для модуля $ARGUMENTS
   - Python API-тесты в `_target-project/autotests/` — найди test-файлы для модуля
   - Определи качество: реальные assertions vs заглушки
   - Построй матрицу: фича → E2E тест → API тест → GitLab тикет

## Перекрёстный анализ

Сравни ВСЕ источники между собой:
- Код ↔ БД: поля есть в модели, но не в таблице (или наоборот)
- Код ↔ Confluence: требование описано, но не реализовано
- Код ↔ Figma: элемент в макете, но не в UI
- Код ↔ Код: обработка ошибок, null-проверки, permissions
- GitLab ↔ Тесты: тикет без автотеста, баг без регрессионного теста
- E2E ↔ API: фича покрыта API-тестом, но нет UI-теста

## Отчёт

Сохрани в `_qa-report-$ARGUMENTS.md` и выведи:
1. Краткая сводка: источники, кол-во проблем
2. Найденные несоответствия (таблица: серьёзность, описание, рекомендация)
3. Сгенерированные тест-кейсы (TC-MODULE-NNN формат, P0→P3)
4. Пробелы в покрытии
5. Рекомендации: что делать дальше

Предложи запустить `/generate-tests $ARGUMENTS` для автоматизации P0/P1 кейсов.
